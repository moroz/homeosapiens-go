## Locked product decisions (2026-06-22, from client Q&A)
- **Recordings are decoupled from events.** Each `video_group` is its **own product** (`video_groups.product_id`). Do NOT reuse an event's product for its recording; no automatic event→recording wiring.
- **Two ways to gain recording access, both end in a `user_product_access` row:**
  - **A. Self-serve purchase — fully automated:** buy a recording's product via existing cart → Stripe → webhook.
  - **B. Manual grant (admin) — the primary flow:** "attended the seminar → here's the recording" applied manually by the client after the event; also covers off-platform/cash payers. Admin inserts `user_product_access` directly, no Stripe.
- **Access source of truth = the `user_product_access` entitlement**, never a live Stripe payment. Recordings gatekept (locked unless entitlement / admin / no-product group). Public free videos live on the Watch page via `videos.is_public`, not as free groups.
- **Events: some paid (Stripe), some free.** Free = one-click RSVP. Paid = cart/Stripe ticket. Event payment grants the event product + registration — does NOT auto-grant a recording.
- **No event capacity limits** (Zoom quota never hit).
- **No auto-refunds** — admin marks refunded → revoke `user_product_access` + cancel registration; money refunded by hand.
- **Stripe stays a direct PSP, NOT a merchant-of-record.** No Paddle/LemonSqueezy. PL/EU pay in PLN; she handles her own PL/EU tax. Indian segment is low-value/mostly free (MoR cost > revenue).
- **Offline payments common** (cash / bank transfer, bypass Stripe) → `orders.payment_method` (stripe/cash/bank_transfer/comp), optional manual `paid_at`; manual grant may create access with no order. Invoicing of offline sales is the client's responsibility, out of app scope.
- **Discount codes: yes** (`product_prices` rule_type `DiscountCode`). EarlyBird optional.
- **Email language follows recipient `preferred_locale`** (PL fallback), not always-PL.
- **Zoom join links deferred** — no static `events.join_url`. Generate per-registrant links via Zoom API later (Meeting Registration API or Server-to-Server OAuth). Until then: in-person events show venue, online events omit the link.

## Code facts worth remembering
- **Event registration: two routes → same `EventRegistrationController.Create`** — `GET /events/:event_id/register` (router.go:128) and `POST /event_registrations/:event_id` (router.go:129). The GET is intentional: it's the post-login redirect target so an anon user who clicks Register is auto-registered after sign-in. Not a bug. `DELETE /event_registrations/:event_id` unregisters. All behind `RequireAuthenticatedUser`.
- **`RegisterBuyerForPaidEvents`** query had a column swap (was `select o.user_id, e.id` into `(event_id, user_id)`); correct is `select e.id, o.user_id`. Possibly add `and e.ends_at > now()` so buying a past event's recording doesn't create a stale registration (undecided).
- **Video schema (decided, not yet migrated):** keep ONE `videos` table (`provider` youtube|cloudfront + `is_public`). Add `description_en`/`description_pl` (Watch page shows descriptions) and a nullable `youtube_id` (stop stuffing YouTube ids into `video_sources.object_key`, which is for CloudFront HLS).
- **Asset variants:** eager, code-defined sizes generated by a batch script (CloudFront stays dumb). Keep `object_key` nullable as override; replace `scaled` bool with `variants_generated_at`; never discard the original; store width/height + extension.
- **Test factories added** in `services/mocks/helpers.go`: `Cart(db, ctx, productIDs...)` and `Order(&OrderInput{DB, Context, Stripe, CartID, Overrides})`.
